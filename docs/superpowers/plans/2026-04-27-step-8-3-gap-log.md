# Step 8.3 — Gap Log

> Running log of gaps surfaced during the dry-run. One entry per gap.
> Severity: blocker / annoying / nice-to-have. Status:

## Gap 1 — Install does not refresh `.agent/` brain on reinstall

- **Severity:** annoying (blocker for tight harness-dev loop, not for users)
- **Surfaced:** Stage 2 verification — sandbox install
- **Symptom:** New `.agent/tools/*.py` and skills added in fork did not
  reach `examples/pdlc-sandbox/` on reinstall. Required manual `cp` to
  test. Same applies to fixes in `.agent/AGENTS.md`, `.agent/protocols/`,
  any updated tool.
- **Root cause:** [harness_manager/install.py:189](../../../harness_manager/install.py)
  copies `.agent/` only when target's `.agent/` does not exist. Once the
  brain is dropped, subsequent installs only refresh adapter-specific
  files listed in `adapter.json`. The "in-place evolution" intent
  (Philosophy A) is correctly preserved for memory + skills, but
  fork-managed tools/protocols/AGENTS.md should refresh.
- **Proposed fix (deferred):** add `--upgrade` flag to install that
  refreshes specific subdirs (`.agent/tools/`, `.agent/protocols/`,
  `.agent/harness/`, `.agent/AGENTS.md`) but never touches
  `.agent/memory/`, `.agent/skills/` (in `in_place` mode), or
  `.claude/agent-memory/`.
- **Status:** deferred-to-8.5 (this is exactly the
  `install.sh --upgrade` work that was already scoped there).
- **Workaround:** manual `cp` for sandbox iteration; fresh installs
  (Stage 3 target at `~/code/case-dryrun-<slug>/`) avoid the issue.

## Gap 2 — `take_install_snapshot` constructed doubled path

- **Severity:** blocker (snapshot capture failed silently in install output)
- **Surfaced:** Stage 2 verification — sandbox install
- **Symptom:** `take_install_snapshot: snapshot_failed —
  ...examples/pdlc-sandbox/examples/pdlc-sandbox/.agent/tools/
  snapshot_diff.py: [Errno 2] No such file or directory`
- **Root cause:** subprocess passed a relative tool path AND
  `cwd=target_root`, so Python re-resolved the path against cwd.
- **Status:** fixed-on-this-branch, commit `1125255`.

## Gap 3 — `document-researcher` ran into PDF + image limits

- **Severity:** annoying (workaround exists; degrades summary quality
  for non-text formats)
- **Surfaced:** Stage 3c — processing 8 uploads
- **Symptom:** `pdftotext` and `pdftoppm` (poppler) not available in
  the local environment; PNG (image) summaries had no auto-extraction
  path. Both were handled defensively (PDF deferred to its `.md`
  sibling; PNG marked duplicate-of-md).
- **Root cause:** `document-researcher` skill assumes UTF-8 readability;
  needs a graceful-degradation rule for binary formats.
- **Proposed fix:** update `document-researcher/SKILL.md` to require
  a fallback summary even when content can't be auto-extracted —
  user-description goes in, file metadata + topics go in, INDEX entry
  is created with `topics: image-only` or `topics: pdf-no-extractor`
  flag. Also document the optional dependencies (`pdftotext`,
  `pdftoppm`, `python-pptx`) in the skill's preconditions.
- **Status:** open (would land as `skill-update: document-researcher`
  via the per-skill self-rewrite hook the skill already documents).

## Gap 4 — sandbox install reinstall blocked by missing install.json

- **Severity:** annoying (reproducibility concern for harness dev loop)
- **Surfaced:** Stage 2 verification — first sandbox reinstall attempt
- **Symptom:** `error: examples/pdlc-sandbox/.agent/ exists but
  install.json does not. this looks like a pre-v0.9 install.` Doctor
  command refused to register in non-interactive shell; required
  manually writing a stub install.json to proceed.
- **Root cause:** doctor flow assumes interactive terminal; no `--yes`
  shortcut for the register-pre-v0.9-adapter case.
- **Proposed fix:** extend `cmd_doctor` to honor `--yes` for the
  register-existing-adapters path, OR have install.sh write a minimal
  install.json stub when it sees the "brain present, no install.json"
  state and `--yes` is set.
- **Status:** open, candidate for Step 8.4 or 8.5.

## Gap 6 — `document-researcher` requires `pypdf` for PDF inputs

- **Severity:** annoying → resolved (`pip install --user pypdf`)
- **Surfaced:** Stage 3c re-run with new files (April 28); 5 of 7 new
  files were PDFs that would have been deferred-to-on-demand under
  the v1 skill rules.
- **Symptom:** With pypdf installed, full-text extraction was clean
  for HarnessX_v4 (28 pages), BOCHK (37 pages), Vince transcript
  (13 pages), Balaji (31 pages), SC AI SDLC (102 pages). Without
  pypdf, all of these would have fallen to "deferred-to-md-sibling"
  treatment — useless for the BOCHK + Balaji decks which had no
  markdown twins.
- **Proposed fix:** add `pypdf` (pure Python, no system deps) to the
  `document-researcher` skill's optional-tools list and document a
  one-liner setup in the skill preconditions: `pip install --user
  pypdf` if PDFs are expected. Better than poppler/pdftotext (system-
  dep, requires brew install).
- **Status:** open as skill-update candidate to document-researcher.

## Gap 7 — Adding a new skill mid-engagement requires manual sync to target

- **Severity:** blocker for tight skill iteration; same root cause as Gap 1

## Gap 5 — `permissions.deny` block correctly fires against installer

- **Severity:** false alarm (turned out to be macOS bash sandbox, not
  a deny-rule misfire)
- **Surfaced:** Stage 2 verification — first install after deny block landed
- **Symptom:** `PermissionError: [Errno 1] Operation not permitted:
  'examples/pdlc-sandbox/.claude/agents/architect.md'` during install.py
- **Resolution:** the deny rules don't apply to Python file writes from
  outside the Claude Code session. The error was the bash sandbox
  blocking writes outside its allowlist. Retried with
  `dangerouslyDisableSandbox: true` and succeeded.
- **Status:** not-a-bug. Documented because the symptom looked like a
  deny-rule problem on first read.

## Gap 9 — Auto-dream candidates dominated by tool-write noise on long sessions

- **Severity:** annoying (low signal-to-noise in graduation pipeline; dilutes review)
- **Surfaced:** Stage 5 post-mortem after Phase 2 deck build (130 episodes, 13 candidates)
- **Symptom:** All 13 staged candidates after Phase 2 are tool-use claims of
  the shape "Wrote storyboard.md (781 lines)" or "Edited content-draft.md:
  replaced X with Y". None are durable cross-engagement lessons. The top
  candidate has priority 891 (66-episode cluster) but its claim is purely
  the file-write event. Net graduations from this run: 0 (correctly — there
  is nothing to graduate).
- **Root cause:** `auto_dream.py` clusters episodes by token overlap of the
  recorded action text. On long content-drafting sessions where the same
  file is written/edited 30+ times, the dominant cluster signal is the
  filename + tool name, not the *insight* embedded in the action. The
  cluster claim then reflects the action shape, not the lesson.
- **Proposed fix (deferred to 8.4):** add an action-type filter to
  `auto_dream.py` so `Write`/`Edit` events on the same file within a single
  session collapse to a single representative episode before clustering.
  Alternatively, weight `memory_reflect.py`-recorded reflections higher
  than tool-action episodes when computing cluster salience — agents
  should reflect explicitly when they want to graduate a lesson, and
  the dream cycle should privilege those over tool-use clusters.
- **Status:** open, candidate for Step 8.4 (auto-dream tuning informed by 8.3 data).

## Gap 10 — Workflow audit ran after storyboard v2, not mid-storyboard

- **Severity:** annoying (rework cost — 6 storyboard restructure moves
  between v2 and v3 that an earlier audit would have caught)
- **Surfaced:** Stage 4 — `audit-framework-lead.md` produced 2026-04-29
- **Symptom:** Framework-lead's 8-section coverage audit fired *after*
  storyboard v2 was complete and identified 3 critical gaps (section 5
  value-gap From-To absent; section 6 MVP mis-framed as scale-out;
  section 8 Investment + next steps entirely missing). v3 then required
  6 structural moves to reconcile against the workflow contract. Each
  move had downstream consequences (renumbering, sticky migration,
  cluster-draft re-scoping). The audit content was correct; the
  *timing* was wrong.
- **Root cause:** `consulting-deck-builder` Phase 1 (Storyboard) does
  not currently include a workflow-contract reconciliation gate before
  marking the storyboard ready for Phase 2. The 8-section coverage
  check is currently an audit step on a completed v2, not a build
  constraint during v1 drafting.
- **Proposed fix:** add a Phase 1.5 gate to `consulting-deck-builder`
  SKILL.md: after storyboard v1 draft, before Phase 2 entry, run an
  8-section coverage check against the source workflow file
  (`final-recommendations-deck.md` or whichever workflow seeded the
  engagement). Each missing or mis-framed section is a v1→v2 fix
  surfaced *before* the storyboard is signed off — not after.
- **Status:** open as skill-update candidate to `consulting-deck-builder`
  (would land via the per-skill self-rewrite hook).

## Gap 11 — `propose_harness_fix.py` never invoked despite engagement-long friction

- **Severity:** annoying (capture mechanism is mechanically present but
  socially invisible — agents don't know to call it)
- **Surfaced:** Stage 5 post-mortem — `HARNESS_FEEDBACK.md` empty after
  130 episodes across Phase 1 + Phase 2 (storyboard build, 5 cluster
  drafts, 4 audits, 3-reviewer panel, final-fix pass)
- **Symptom:** Phase 2 surfaced multiple harness-shaped frictions that
  *should* have produced HARNESS_FEEDBACK entries but didn't:
  - Workflow audit timing (Gap 10) — discovered by framework-lead;
    not captured as a harness-fix proposal
  - Cold-transition findings on Slides 8→9, 11→12, 15→16 — pattern-
    shaped, would have generated a `consulting-deck-builder` skill
    update (add explicit cold-transition pass to Phase 2 audit step)
  - Agent-memory only populated for 2 of ~7 active agents — capture
    rule on when agents *should* persist project/feedback memories
    is undefined
- **Root cause:** `propose_harness_fix.py` exists in `.agent/tools/` and
  is documented in `adapters/claude-code/CLAUDE.md`, but no skill or
  protocol **names the trigger** for invocation. Agents have no rule
  of the form "when you hit harness friction, call this." The tool is
  invisible without a behavioral hook.
- **Proposed fix:** two-part wiring —
  1. Add to `adapters/claude-code/CLAUDE.md` "When to use" block: an
     explicit list of trigger conditions (e.g., "skill missing a step",
     "workflow ordered wrong", "agent memory sparse", "audit found a
     pattern-shaped fix").
  2. Add a turn-end stop-hook (or session-end hook) that prompts the
     agent: "Did you encounter any harness-shaped friction this session?
     If so, capture it now via `propose_harness_fix.py`." Forces the
     reflection pass before the session closes.
- **Status:** open, candidate for Step 8.4 (paired with Gap 9 — both are
  about closing the harness-learning capture loop).

## Gap 8 — No structural-conformance enforcement on skills

- **Severity:** annoying — caused 5/26 skills to silently miss the
  self-rewrite hook for months until audited
- **Surfaced:** mid-engagement Pulkit question ("are we employing
  all the items we're trying to create for proper harness work?")
- **Symptom:** skillforge constraint says "always include self-rewrite
  hook" but no automated check enforced it. Result: `analysis`,
  `context-search`, `document-assembly`, `draft-status-update`,
  `review` shipped without the hook. Convention drifted silently.
- **Root cause:** the harness has memory + episodic + dream cycle for
  *engagement* learning, but no inward-pointing validators for its
  own development. Convention is documentation, not enforcement.
- **Fix landed on this branch (commit `dc5fa20`):**
  - Added 5 missing self-rewrite hooks
  - New `.agent/tools/skill_linter.py` validating frontmatter shape,
    self-rewrite hook presence, manifest match, index match
  - New `git-hooks/pre-commit` runs linter on staged skill changes
  - Local activation: `git config core.hooksPath git-hooks`
- **Status:** ✅ closed-on-this-branch. Linter passes 26/26.
  - Follow-up for 8.4: extend linter to check more conventions
    (trigger phrasing, constraint shape, examples block presence)
  - Follow-up for 8.4: consider `.pre-commit-config.yaml` for
